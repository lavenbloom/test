PHASE 5: TROUBLESHOOTING & FIXES
-------------------------------------------------------------
[ERROR 1: Remote IP Accessibility]
- Issue: `REACT_APP_API_URL` defaulted to `http://localhost:8080`, causing requests from the EC2 instance's IP to fail because the browser resolved `localhost` to the user's laptop.
- Fix: Modified `docker-compose.yml` to set `REACT_APP_API_URL=""` and updated `api.ts` to fallback to an empty string. This allows Axios to use relative URLs (matching the NGINX gateway origin natively).
- Fix: Modified `nginx-gateway/nginx.conf` to use `server_name _;` instead of `localhost` to catch all external EC2 traffic.

[ERROR 2: Missing Python Dependencies]
- Issue: Pydantic `v2` `EmailStr` in Auth Service required the `email-validator` library to run.
- Fix: Added `email-validator==2.1.0.post1` to `auth-service/requirements.txt`.

[ERROR 3: Frontend TypeScript Build Failure]
- Issue: Docker build failed on `npm run build` because `tsc` failed in strict mode due to unused variables and missing Vite types.
  - TS6196: 'Habit' is declared but never used.
  - TS6133: 'React' is declared but its value is never read.
  - TS2339: Property 'env' does not exist on type 'ImportMeta'.
- Fix: Removed unused `interface Habit` in `HabitGrid.tsx`.
- Fix: Removed unused `import React from 'react';` in `Dashboard.tsx`.
- Fix: Created `src/vite-env.d.ts` containing `/// <reference types="vite/client" />` to expose `import.meta.env` definitions.
- Fix: Removed the invalid `"references": [{ "path": "./tsconfig.node.json" }]` link in `tsconfig.json` which would have caused compilation to crash.

[ERROR 4: Performance N+1 API Calls]
- Issue: `HabitGrid.tsx` sent a separate log-fetch request per habit inside a `Promise.all` `.map` loop, creating N+1 redundant queries.
- Fix: Optimized `fetchHabits` to fetch `/habit/habits` and `/habit/habits/logs` simultaneously in one `Promise.all` and map the values in memory.

PHASE 8: AUTHENTICATION FIXES
-------------------------------------------------------------
[STEPS TAKEN]
- User reported registration failure on the EC2 instance.
- Diagnosed that `passlib[bcrypt]` in `auth-service` dynamically installs `bcrypt >= 4.0`, which contains breaking changes that cause `passlib`'s hashing functions to crash with an `AttributeError`.
- Fix: Pinned `bcrypt==3.2.2` in `auth-service/requirements.txt` to restore compatibility.
- Diagnosed that the Nginx API Gateway container had crashed because it was started while backend services were still building, leading to a host resolution error. Because the Gateway was dead, the UI fallback behavior caused it to query the frontend container on port 3000, failing all API requests.
- Fix: Modified user's `auth-service/Dockerfile` to include `gcc python3-dev libffi-dev` alongside `libpq-dev` to ensure `bcrypt` and database drivers compile successfully from source on slim images.

-------------------------------------------------------------
PHASE 9: DOCKERFILE STANDARDIZATION
-------------------------------------------------------------
[STEPS TAKEN]
- User confirmed that converting auth-service Dockerfile from multi-stage to single-stage fixed container startup.
- Root cause of multi-stage failure: `pip install --user` in the builder stage copies packages to `/root/.local`, but the `COPY --from=builder /root/.local /root/.local` step is brittle on slim images because `uvicorn` and other binaries end up on a PATH that may not be set correctly for non-root users.
- Fix applied to habit-service/Dockerfile: Replaced multi-stage build with identical single-stage pattern as working auth-service.
- Fix applied to journal-service/Dockerfile: Same single-stage conversion.
- Fix applied to notification-service/Dockerfile: Same single-stage conversion.
- All three Dockerfiles now use: `python:3.11-slim` base, `gcc python3-dev libffi-dev libpq-dev` build deps, `pip install --no-cache-dir`, non-root `appuser`, and `uvicorn` CMD.
